🔧 5. Problemas Detectados y Soluciones (How-To para Desarrolladores)

🔴 PROBLEMA 1: LCP Crítico – Largest Contentful Paint > 4s (Móvil)

Impacto: 2.997+ URLs con LCP malo. El elemento LCP principal tarda hasta 18s en laboratorio.

URLs más afectadas: /marcas/poly (6,3s CrUX), /productos/* (4,4s), / homepage (4,8s)

Causa raíz: TTFB de 3,5s + CSS y JS bloqueantes + imágenes no optimizadas + sin preload de elemento LCP

✅ Solución 1.1: Reducir TTFB – Mejorar tiempo de respuesta del servidor

1
Implementar Full Page Cache (FPC) agresivo en Magento

En el admin de Magento: Stores → Configuration → Advanced → System → Full Page Cache. Activar Varnish Cache. Asegurarse de que las páginas de categoría y producto tengan un TTL mínimo de 3600s.

2
Configurar Varnish/CDN correctamente

Verificar que las cabeceras Cache-Control sean correctas. El TTFB de 3,5s indica que muchas peticiones no están siendo servidas desde caché. Revisar las exclusiones de Varnish (cookies de sesión que invalidan la caché).

# En Varnish VCL, verificar que páginas de categoría no sean invalidadas innecesariamente
sub vcl_recv {
  if (req.url ~ "^/(productos|marcas|walkie-talkie|telefonos)" && !req.http.Cookie ~ "X-Magento-Cache-Debug") {
    unset req.http.Cookie;
  }
}
3
Usar un CDN con edge locations en España/Europa

Si no hay CDN: implementar Cloudflare (plan gratuito/Pro) o AWS CloudFront. El objetivo es servir HTML desde ubicaciones cercanas al usuario y cachear assets estáticos con TTL largo.

✅ Solución 1.2: Eliminar CSS bloqueante del renderizado

1
Detectar CSS crítico (above-the-fold)

El archivo styles-m.css (87KB) bloquea 1.350ms el renderizado. Extraer el CSS crítico (above the fold) e insertarlo inline en el <head>.

2
Cargar el CSS no crítico de forma asíncrona en Magento 2
<!-- En layout XML del tema (default_head_blocks.xml) -->
<head>
  <css src="css/styles-critical.css" />
  <link rel="preload" as="style" href="css/styles-m.css" 
        onload="this.rel='stylesheet'" />
  <noscript><link rel="stylesheet" href="css/styles-m.css" /></noscript>
</head>
3
Eliminar CSS no utilizado (96KB de ahorro)

Usar PurgeCSS o herramientas similares para identificar y eliminar reglas CSS no utilizadas en styles-m.css. El 93% del archivo es código no usado en la mayoría de páginas.

# Instalar PurgeCSS como dependencia de desarrollo
npm install --save-dev @fullhuman/postcss-purgecss

# Configurar en postcss.config.js
module.exports = {
  plugins: [
    require('@fullhuman/postcss-purgecss')({
      content: ['./app/design/**/*.phtml', './app/design/**/*.html'],
      safelist: [/^slick-/, /^owl-/, /^mage-/]
    })
  ]
}

✅ Solución 1.3: Preload del elemento LCP (imagen principal)

1
Identificar el elemento LCP en cada tipo de página

En páginas de producto y marca, el LCP suele ser la imagen principal del producto o el hero banner. Verificar con Chrome DevTools → Performance → LCP Element.

2
Agregar rel="preload" para la imagen LCP
<!-- En el layout XML o en el bloque head de Magento 2 -->
<link rel="preload" as="image" 
      href="/media/catalog/product/cache/[hash]/imagen-lcp.jpg"
      fetchpriority="high" />

<!-- O directamente en la imagen HTML -->
<img src="imagen-lcp.jpg" 
     fetchpriority="high" 
     loading="eager"
     width="800" height="800"
     alt="Descripción producto" />
3
En Magento 2: modificar el bloque de imagen de producto

En el archivo Magento_Catalog/templates/product/image_with_borders.phtml o similar, agregar fetchpriority="high" y loading="eager" a la imagen principal (la primera imagen visible).

🔴 PROBLEMA 2: CLS Crítico – Cumulative Layout Shift > 0,25 (Móvil)

Impacto: 3.180 URLs con CLS malo. Homepage con CLS de 0,42 (muy crítico). El mayor causante es una imagen sin dimensiones.

Elemento causante principal: img.img-fluid (imágenes de categorías y marca sin width/height declarados)

✅ Solución 2.1: Declarar dimensiones explícitas en todas las imágenes

1
Regla global CSS para reservar espacio en imágenes
/* En styles-m.css o styles-critical.css */
img {
  width: 100%;
  height: auto;
}

/* Para imágenes de producto en listados */
.product-image-wrapper {
  position: relative;
  padding-bottom: 100%; /* Ratio 1:1 para imágenes cuadradas */
  overflow: hidden;
}
.product-image-photo {
  position: absolute;
  top: 0; left: 0;
  width: 100%;
  height: 100%;
  object-fit: contain;
}
2
Agregar width y height a todas las imágenes en templates Magento

En todos los archivos .phtml de templates de imágenes, asegurarse de que el tag <img> incluya los atributos width y height:

<!-- ANTES (incorrecto) -->
<img class="img-fluid" src="getImageUrl() ?>" alt="..." />

<!-- DESPUÉS (correcto) -->
<img class="img-fluid" 
     src="getImageUrl() ?>" 
     alt="..."
     width="getWidth() ?>"
     height="getHeight() ?>" />
3
Corregir específicamente la imagen de hero en páginas de marca

El elemento causante principal es <img class="img-fluid rounded-lg" src="/media/upload_img/common/brand/poly-lifestyle.webp">. Debe tener dimensiones explícitas:

<img class="img-fluid rounded-lg" 
     src="/media/upload_img/common/brand/poly-lifestyle.webp"
     alt="Poly" 
     width="435" height="300"
     loading="lazy" />
4
Revisar el header/switcher de idioma

El div.header > div.switcher también causa pequeños CLS. Verificar que tenga altura reservada con CSS y no cambie de tamaño al cargar.

✅ Solución 2.2: Fuentes web sin FOUT (Flash Of Unstyled Text)

1
Precargar fuente Mulish y usar font-display: optional
<!-- En el head del HTML o layout XML de Magento -->
<link rel="preload" as="font" type="font/woff2"
      href="https://fonts.gstatic.com/s/mulish/v18/1Ptvg83HX_SGhgqk3wotYKNnBQ.woff2"
      crossorigin="anonymous" />

<!-- En CSS -->
@font-face {
  font-family: 'Mulish';
  font-display: optional; /* Evita CLS por FOUT */
  src: url('...woff2') format('woff2');
}

🟠 PROBLEMA 3: INP Alto – Interaction to Next Paint > 200ms (Móvil)

Impacto: 3.082 URLs con INP que necesita mejora. Páginas de telefonía móvil con INP de 412ms (crítico).

Causa raíz: Hilo principal bloqueado 10,7s total · RequireJS eval: 3.419ms · LiveChat: 1.962ms · SST scripts: 688ms

✅ Solución 3.1: Reducir JavaScript que bloquea el hilo principal

1
Auditar y diferir scripts de terceros no críticos

LiveChat, OneTrust, Bing Ads, Piwik Pro y scripts SST suman más de 4.000ms de CPU. Cargarlos de forma diferida con async/defer o mediante Intersection Observer cuando el usuario hace scroll.

<!-- Cargar LiveChat solo cuando el usuario interactúa -->
<script>
document.addEventListener('DOMContentLoaded', function() {
  // Diferir LiveChat 3 segundos o hasta primera interacción
  let loaded = false;
  const loadLiveChat = () => {
    if (loaded) return;
    loaded = true;
    const s = document.createElement('script');
    s.src = 'https://cdn.livechatinc.com/tracking.js';
    s.async = true;
    document.head.appendChild(s);
  };
  
  setTimeout(loadLiveChat, 3000);
  document.addEventListener('mousemove', loadLiveChat, {once: true});
  document.addEventListener('touchstart', loadLiveChat, {once: true});
  document.addEventListener('keydown', loadLiveChat, {once: true});
});
</script>
2
Optimizar RequireJS en Magento 2 – Migrar a módulos ES6 o mejorar bundling

RequireJS está usando 3.419ms de CPU. Esto indica que el bundle de Magepack no está correctamente configurado o hay demasiados módulos cargándose en el critical path.

# Regenerar bundles de Magepack con configuración optimizada
php bin/magento setup:static-content:deploy es_ES -f
php bin/magento cache:clean

# Verificar configuración de magepack en magepack.config.js
# Asegurarse de que bundle-common solo incluye módulos realmente comunes
3
Corregir forced reflows de jQuery

jQuery está causando 185-261ms de redistribución forzada. Esto ocurre cuando JS lee propiedades geométricas (offsetWidth, getBoundingClientRect) y luego modifica el DOM, forzando al navegador a recalcular el layout.

// MALO: Leer y escribir DOM alternadamente (forced reflow)
elements.forEach(el => {
  const height = el.offsetHeight; // Lee → fuerza reflow
  el.style.height = height + 'px'; // Escribe
});

// CORRECTO: Agrupar lecturas primero, luego escrituras
const heights = Array.from(elements).map(el => el.offsetHeight); // Lee todo
elements.forEach((el, i) => {
  el.style.height = heights[i] + 'px'; // Escribe todo
});
4
Implementar web workers para tareas pesadas de JS

Mover lógica de procesamiento no relacionada con el DOM a Web Workers para liberar el hilo principal.

🟠 PROBLEMA 4: Exceso de JavaScript – 1.493 KiB Sin Usar

Causa: El bundle bundle-common.min.js de Magepack pesa 1.562KB de los cuales 1.045KB (67%) no se usa en la página de marca.

✅ Solución 4.1: Reconfigurar Magepack para bundling más granular

Crear bundles específicos por tipo de página

🟠 PROBLEMA 5: Imágenes No Optimizadas – 698 KiB de Ahorro

Causas: (1) Imágenes JPEG/PNG en lugar de WebP/AVIF. (2) Imágenes de 800x800px servidas a 100-150px de display. (3) Sin responsive images con srcset.

✅ Solución 5.1: Convertir imágenes a WebP/AVIF en Magento 2

1
Instalar extensión de optimización de imágenes o configurar el procesado en el servidor
# Opción 1: Usar el módulo de Magento 2 para WebP (disponible desde Magento 2.4.x)
# Verificar en vendor/magento/module-catalog/etc/config.xml que WebP está habilitado

# Opción 2: Configurar Nginx para servir WebP automáticamente
# En nginx.conf:
location ~* .(jpg|jpeg|png)$ {
    set $webp_suffix "";
    if ($http_accept ~* "webp") {
        set $webp_suffix ".webp";
    }
    if (-f "$request_filename$webp_suffix") {
        rewrite (.*) $1$webp_suffix break;
    }
}
2
Implementar responsive images con srcset en los templates de Magento
<!-- En Magento_Catalog/templates/product/list/item.phtml -->
<img
  src="getImageUrl('category_page_list', $product) ?>"
  srcset="
    getImageUrl('category_page_list_small', $product) ?> 300w,
    getImageUrl('category_page_list', $product) ?> 600w,
    getImageUrl('category_page_list_large', $product) ?> 1200w"
  sizes="(max-width: 768px) 150px, (max-width: 1024px) 200px, 264px"
  width="264" height="264"
  loading="lazy"
  alt="escapeHtmlAttr($product->getName()) ?>" />
3
Configurar tamaños de imagen correctos en Magento

En Stores → Configuration → Advanced → Catalog → Product Image Settings, ajustar los tamaños de caché de imágenes para que coincidan con las dimensiones reales de display:

# En view.xml del tema (app/design/frontend/Onedirect/indigo/etc/view.xml)
<image id="category_page_list" type="small_image">
  <width>264</width>
  <height>264</height>
</image>

# Regenerar imágenes tras cambiar tamaños:
php bin/magento catalog:images:resize

🔴 PROBLEMA 6: TTFB Crítico – 3,5 segundos

Impacto: TTFB de 3,5s afecta directamente al LCP y FCP. El objetivo es <0,8s (bueno) o al menos <1,8s.

Causa probable: PHP lento, Magento sin caché de opcode, consultas DB no optimizadas, o FPC no funcionando correctamente.

✅ Solución 6.1: Optimizar la pila PHP/MySQL de Magento

1
Verificar y optimizar OPCache de PHP
# En php.ini:
opcache.enable=1
opcache.memory_consumption=512
opcache.interned_strings_buffer=64
opcache.max_accelerated_files=60000
opcache.revalidate_freq=2
opcache.save_comments=1
opcache.enable_cli=1
2
Activar Redis para caché de sesiones y caché de configuración
# app/etc/env.php - Configurar Redis para caché
'session' => [
  'save' => 'redis',
  'redis' => [
    'host' => '127.0.0.1',
    'port' => '6379',
    'password' => '',
    'timeout' => '2.5',
    'persistent_identifier' => '',
    'database' => '2',
    'compression_threshold' => '2048',
    'compression_library' => 'gzip',
  ]
],
'cache' => [
  'frontend' => [
    'default' => ['backend' => 'MagentoFrameworkCacheBackendRedis', ...],
    'page_cache' => ['backend' => 'MagentoFrameworkCacheBackendRedis', ...]
  ]
]
3
Implementar Elasticsearch/OpenSearch correctamente

Para búsquedas y filtros de categoría, Elasticsearch mal configurado puede ser causa de TTFB alto en páginas de listado. Verificar que las consultas de catálogo usen el índice de ES y no consultas directas a MySQL.
